2. Modele danych (Prisma + Postgres)
Zbuduj plik prisma/schema.prisma z modelami:
User – standard (id, email, name, role: STUDENT | ADMIN)
Book – id, userId, title, status: UPLOADED | PROCESSING | PROCESSED | FAILED, createdAt, updatedAt
BookPage – id, bookId, pageNumber, rawText, ocrMeta
BookChunk – id, bookId, pageId, type: TASK | THEORY | EXAMPLE | OTHER, content, position, createdAt
Embedding – id, chunkId, vector (pgvector), provider
GraphNode – id, bookId, chunkId?, type: CONCEPT | TASK | SECTION, label
GraphEdge – id, bookId, sourceId, targetId, type: SIMILAR | DEPENDS_ON | NEXT | PART_OF, weight
ChatSession – id, userId, bookId, createdAt, endedAt?
ChatMessage – id, sessionId, role: USER | ASSISTANT, content, createdAt, avatarVideoUrl?
Dodaj relacje, indeksy i typ Vector dla embeddingów (pgvector).
3. Serwisy domenowe – szkielety i logika
Zaimplementuj w TypeScript klasowe serwisy (zależności przez konstruktor – dependency injection). Każdy serwis ma tylko logikę domenową, bez szczegółów HTTP.
BookIngestionService
createBook(userId, fileMeta) → tworzy rekord Book ze statusem UPLOADED.
processBook(bookId, filePath) → woła OcrAdapter (na razie TODO) i zapisuje BookPage; potem wywołuje BookChunkingService i EmbeddingService, ustawia status PROCESSED.

BookChunkingService
Heurystycznie dzieli BookPage.rawText na BookChunk (wykrywa nagłówki „Zadanie”, „Przykład”, itp.).

EmbeddingService
Batchuje BookChunk i woła EmbeddingAdapter.embed(texts), zapisuje wektory w Embedding.

GraphBuilderService
Na podstawie BookChunk + Embedding tworzy GraphNode i GraphEdge.
Używa prostej logiki:
tworzy węzły dla wszystkich chunków,
liczy podobieństwa wektorowe i dodaje krawędzie SIMILAR powyżej progu,
dodaje strukturalne krawędzie NEXT (kolejne strony/chunki) i PART_OF (sekcja → zadania).


KnowledgeQueryService
getContextForQuestion(bookId, question, selectedNodeId?)
jeśli jest selectedNodeId, bierze węzeł + sąsiadów i ich chunk.
robi wektorowe wyszukiwanie na Embedding po pytaniu (RAG).
łączy, deduplikuje i zwraca listę fragmentów jako kontekst.


TutoringService
startSession(userId, bookId)
handleUserMessage(sessionId, message, selectedNodeId?)
pobiera historię sesji,
korzysta z KnowledgeQueryService,
buduje prompt (rola: nauczyciel, nie podawaj od razu odpowiedzi, zadawaj pytania pośrednie),
woła LlmAdapter.complete(prompt),
zapisuje ChatMessage (USER, potem ASSISTANT) i zwraca odpowiedź.


AvatarService
generateClipForMessage(chatMessageId)
pobiera ChatMessage tutora,
woła AvatarAdapter.createClip({ text, avatarId }),
zapisuje avatarVideoUrl w ChatMessage.


W serwisach nie używaj bezpośrednio Prisma – korzystaj z repozytoriów.

Zdefiniuj interfejsy i proste implementacje (z TODO tam, gdzie trzeba klucze API):

Adaptery nie powinny znać modeli Prisma – tylko prosty tekst/parametry.
5. API routes – połączenie front ↔ backend
Zaimplementuj następujące Route Handlers (bez nadmiernego skomplikowania, ale w pełni działające):
POST /api/books
przyjmuje multipart/form-data z plikiem PDF lub obrazami.
tworzy Book (BookIngestionService.createBook), zapisuje plik (np. do uploads/ lub tymczasowo),
uruchamia w tle (na razie await – później do przeniesienia do job queue) processBook,
zwraca bookId.

GET /api/books
zwraca książki zalogowanego użytkownika.

GET /api/books/[bookId]/graph
zwraca węzły i krawędzie grafu jako JSON (lista nodes, edges).

POST /api/chat
body: { sessionId?, bookId, message, selectedNodeId? }
jeśli brak sessionId, tworzy nową sesję przez TutoringService.startSession.
woła TutoringService.handleUserMessage,
zwraca { sessionId, assistantMessage, mainNodeId, avatarVideoUrl? }.

POST /api/avatar/clip
body: { chatMessageId }
woła AvatarService.generateClipForMessage,
zwraca videoUrl.

Do walidacji wejścia użyj zod.
